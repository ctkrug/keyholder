# Architecture

A map of the codebase for whoever (or whatever) picks this up next. See `docs/VISION.md` for
why it's built this way and `docs/DESIGN.md` for the visual system.

## Data flow

```
schema.js (SECTIONS)
    │
    ├──> form.js::renderForm(container, SECTIONS, state, onChange)
    │        builds the DOM form, mutates `state` in place on every input,
    │        calls onChange() after each change
    │
    ├──> model.js::createInitialState(SECTIONS)     -> the mutable `state` object
    ├──> model.js::buildDocument(SECTIONS, state)    -> the document model
    ├──> model.js::buildMeta(state.basics)           -> { title, subtitle }
    ├──> model.js::applyDraft(SECTIONS, state, draft) -> merges a loaded draft into `state`
    │        (storage.js::loadDraft supplies `draft`; sanitizes/ignores anything
    │         that no longer matches the current schema shape)
    │
    ├──> preview.js::documentToHtml(doc, meta)  -> HTML string  -> #preview.innerHTML
    ├──> text.js::documentToText(doc, meta)     -> plain text (unused by app.js yet;
    │                                               kept for a future "copy as text" feature)
    └──> pdf-layout.js::documentToPdfLines(doc, meta) -> styled instruction list
             └──> pdf.js::exportPdf(doc, meta)        -> walks the instructions with jsPDF,
                                                          paginates, triggers a file download
```

`app.js` is the only DOM-touching glue: it owns `state`, wires `renderForm`'s onChange to
re-run `buildDocument` + `documentToHtml`, and wires the Export button to `exportPdf`. Everything
it calls is a pure function over plain data — that's what makes the rest unit-testable without a
DOM (`node --test`, no jsdom dependency).

## State shape

`state` is namespaced by section id, matching `SECTIONS` in `schema.js`:

```js
state[sectionId] = { fieldId: value, ... }        // fixed section (e.g. basics, notes)
state[sectionId] = [ { fieldId: value }, ... ]     // repeatable section (e.g. accounts)
```

`model.js` is the only module that understands this shape's rules: a repeatable entry is
"empty" when every field is blank/whitespace (`entryIsEmpty`), blank entries are dropped from
the rendered document, and a repeatable section with nothing left renders `empty: 'None listed'`
instead of zero groups.

## Files

| File | Role |
|---|---|
| `src/js/schema.js` | The single source of truth for checklist structure — section id, title, description, fields, and whether a section is repeatable. Adding a section/field here flows through form, preview, and PDF with no other code change. |
| `src/js/model.js` | Pure transforms: schema + state → initial state / document model / masthead meta. No DOM. |
| `src/js/form.js` | DOM-touching: builds form controls from the schema, binds them to `state`, manages repeatable add/remove. Exports `controlId` (pure, tested) for stable/collision-free element ids. |
| `src/js/preview.js` | Pure: document model → escaped HTML string for the live preview "paper sheet". |
| `src/js/text.js` | Pure: document model → plain text (used by nothing yet — a text-export or copy-to-clipboard feature would reuse this). |
| `src/js/pdf-layout.js` | Pure: document model → ordered list of `{ style, text }` instructions. Keeps section/field ordering and text content unit-testable separately from jsPDF/font-metric concerns. |
| `src/js/pdf.js` | The only module touching the vendored jsPDF global. Walks `pdf-layout.js`'s instructions, applies the paper-and-ink styles, wraps text, paginates, stamps footers, and calls `.save()`. |
| `src/js/storage.js` | Optional `localStorage` draft persistence. Takes an injectable `storage` argument instead of reaching for `window.localStorage` directly, so it unit-tests with a plain fake. Autosave defaults to off; turning it off clears the draft immediately. |
| `src/js/app.js` | Wires everything above to the DOM: owns `state`, re-renders the preview on every input, handles Export PDF (press/success/error states), the autosave toggle (load/save/clear draft), and Clear-all (confirm + full state reset). |
| `src/css/style.css` | The whole design system: tokens, vendored `@font-face`, layout grid, form control states, buttons, the preview "paper" styling, responsive breakpoints, and `@media print`. |
| `src/vendor/` | Gitignored, regenerated by `scripts/copy-vendor.js` (runs on `npm install` via `postinstall`). Copies jsPDF and the Fraunces/Inter woff2 files out of `node_modules` so nothing is fetched from a CDN at runtime. |
| `scripts/copy-vendor.js` | The vendoring step described above. |

## Running it

```bash
npm install       # also runs postinstall -> populates src/vendor/
npm test          # node --test test/ — pure-function unit tests, no browser needed
npm run lint      # eslint src test scripts
```

To use the app itself: serve `src/` with any static file server (`npx serve src`,
`python3 -m http.server --directory src`, etc.) and open it in a browser. Opening
`src/index.html` directly via a `file://` URL does **not** work — browsers block ES module
imports (`<script type="module">`) from `file://`, so `app.js` never runs and the form/preview
stay empty. This isn't a bug to fix; it's an inherent `file://` restriction, which is why the
README's dev instructions say "serve" rather than "open."
